# Decisions

Running log of implementation choices and their tradeoffs (see CMS_SYSTEM_SPEC.md §10).

## Phase 0

- **TypeScript pinned to ~5.9 instead of 7.x.** TS 7 (the Go-based compiler) is current on npm,
  but editor tooling and the Next.js TS plugin ecosystem still target 5.x. Revisit once Next
  officially supports 7.
- **Workspace packages ship raw TypeScript** (`main: src/index.ts`) and apps consume them via
  `transpilePackages` — no per-package build step, so `pnpm build` only builds the two apps.
  Simplest setup for a private monorepo; add a bundler only if a package ever needs publishing.
- **Tailwind v4** (CSS-first config). Shared semantic tokens live in
  `packages/config/tailwind/theme.css` as `@theme` variables; the per-site `tokens.json` →
  theme generation pipeline lands in Phase 2.

## Phase 1

- **Column-level restrictions via BEFORE UPDATE triggers, not RLS.** RLS is row-level only;
  the spec's "client_editor may update only `sections.props` / `pages.seo`" needs column
  granularity. Both admin roles map to the same Postgres role (`authenticated`), so column
  grants can't distinguish them. A trigger compares OLD/NEW and raises `42501` for editors
  touching restricted columns. Alternative (a `security definer` RPC per edit operation) was
  rejected as heavier for Phase 3's auto-generated forms.
- **client_editor column set is strictly per spec: `sections.props` and `pages.seo`.**
  Editors cannot toggle page/section `status` (publishing stays a studio action for now).
  Loosen the trigger if clients need self-serve publishing (revisit at Phase 4).
- **Explicit DML grants in the migration.** This local stack's default privileges don't give
  `authenticated`/`service_role` DML on new tables, so the migration grants
  select/insert/update/delete explicitly. `anon` gets nothing — public content is served
  exclusively through the API-key-checked `/api/content/[siteSlug]` route (spec §4 chose the
  API-route approach over `get_published_site()`). Future migrations must repeat grants for
  new tables.
- **API keys stored as SHA-256 hex** of a high-entropy key; the route compares digests with
  `crypto.timingSafeEqual`. Unknown slug and bad key both return 401 so slugs don't leak.
- **Seed uses a fixed local dev key** (`local-dev-demo-content-key`) so acceptance tests are
  reproducible; real per-site keys are generated by `create-site.ts` in Phase 5.
